# feat: 应用 UI 展示名称调整为 `Ledgerly`

**公用约定**：以 `.cursor/prd/agent-dev-guide-summary.md` 为准。仓库级 Agent 基线（Cursor 始终应用）：`.cursor/rules/creator-cfo-always.mdc`。

## 平台描述

- 当前项目为 **Expo + React Native + Expo Router** 的移动端本地优先产品。
- 目前工程名或默认配置中仍存在 `creator-cfo-monorepo` 这一偏开发态的名称。
- 本需求的目标是把**UI 层面用户可见的应用名称**统一为 **Ledgerly**，让产品展示更贴近正式形态。

## 背景与目标

- `creator-cfo-monorepo` 更像仓库名或开发代号，不适合作为最终产品展示名。
- 如果需求只写成“把 app name 改成 Ledgerly”，实现时很容易遗漏：
  - 只改了一个配置文件，其他展示位仍显示旧名
  - UI 展示位改了，但技术标识也被误改
  - 把“展示名调整”误做成“工程重命名”
- 本 PRD 的目标是让 agent 明确：这次需求是**只改 UI / 展示层名称**，不是重命名工程、包名、bundle id 或其他技术配置。

## 存储与契约影响

- **结构化 DB / 文件仓**：默认**无影响**。
- **设备态 / 本地状态**：默认**无新增字段**、**无迁移**。
- **须同步评估的工件**：
  - [ ] `packages/storage/src/contracts.ts`
  - [ ] `docs/contracts/`
  - [ ] `packages/schemas/src/index.ts`
  - [x] 自动化测试或基础构建验证
  - [x] `tests/smoke/README.md`（如有名称相关展示路径）
- **说明**：
  - 本需求原则上不应触发 storage/schema 变更。
  - 如果实现过程中发现旧名称同时出现在 UI 文案和技术配置中，只处理**用户可见 UI 展示位**，不要误扩展到工程或原生标识。

## 一句话需求

- 将应用中**用户可见的 UI 展示名称**从 `creator-cfo-monorepo` 调整为 `Ledgerly`，且不改动任何技术标识。

## 需求描述

### 1. 用户可见 UI 名称统一为 `Ledgerly`

- App 中用户可见的 UI 名称应统一为 `Ledgerly`。
- 至少需要覆盖以下典型展示位：
  - 应用内页面标题、导航标题、空状态、设置页或其他直接展示给用户的名称文案
  - 如当前 UI 中存在明显暴露 `creator-cfo-monorepo` 的位置，应同步替换
- 目标是让用户看到的名字统一，不再暴露仓库名 `creator-cfo-monorepo`。

### 2. 只改 UI 层，不改技术配置

- 本需求默认只处理**用户可见 UI 文案**。
- 非目标行为：
  - 不重命名 monorepo 根目录
  - 不修改 npm package 名称
  - 不修改 Expo `slug`
  - 不修改 bundle identifier / applicationId / package name
  - 不修改原生工程名、workspace 名、target 名
  - 不借本需求顺带改动品牌文案、Logo、域名或营销物料

### 3. 展示一致性要求

- 名称修改后，UI 展示不应出现“一个地方叫 Ledgerly，另一个地方仍是 creator-cfo-monorepo”的割裂情况。
- 需要优先排查应用内直接展示给用户的名称来源，而不是全局替换所有字符串。
- 若某些字段虽然也包含旧名，但属于技术标识或构建配置，应明确保留，不要改动。

### 4. 回归与风险控制

- 名称修改后，应确保：
  - UI 文案展示符合预期
  - 本地启动与现有流程未被改坏
  - 不会因为误改技术标识导致构建或启动异常
- 如果旧名称仍保留在技术字段中，属于预期行为，应在实现说明中明确写清“为什么不改”。

## 范围边界

**包含**：

- 应用内用户可见名称改为 `Ledgerly`
- 评估并同步 UI 展示位
- 补齐至少一项 UI 或启动验证
- 如有必要，补充 smoke 中对名称展示的人工验证说明

**不包含**：

- 重命名仓库、workspace、package 名称
- 修改 bundle id / package name / slug 等技术标识
- 品牌升级、Logo 替换、视觉系统重做
- 重构原生工程结构或发布流程

## 实现约束

- 必须优先区分“用户可见 UI 文案”和“技术标识符”，不要混改。
- 优先做最小闭环：
  - 找出当前 UI 中旧名称的来源
  - 修改用户可见展示名
  - 检查页面展示一致性
  - 跑基础验证
- 如果一个字段会影响构建、安装包标识、深链、更新渠道或原生发布配置，一律不要改。
- 若 Native / Web 展示位不同，需在实现说明里写明此次只覆盖哪些 UI 展示位。

## 验收标准

- [ ] 用户可见的 UI 名称已从 `creator-cfo-monorepo` 调整为 `Ledgerly`。
- [ ] 主要 UI 展示位中不出现明显的旧名残留。
- [ ] 本次改动没有改动 bundle id、package name、slug 或其他技术标识。
- [ ] 至少完成一次基础验证，例如启动检查、UI 检查或 smoke 验证。
- [ ] 如果旧名称仍保留在技术字段中，PR 中已明确说明“这是预期保留，不属于本次范围”。
- [ ] 如有名称相关的 smoke 路径，已补充或更新说明。

## 建议测试点

- 检查应用内标题、设置页、欢迎页或其他用户可见位置是否显示 `Ledgerly`。
- 检查 UI 中是否仍残留 `creator-cfo-monorepo`。
- 检查本次改动后项目是否仍可正常启动。
- 检查 bundle id / package name / slug 等技术字段是否未被改动。

## Testor 验收补充

- 本需求虽偏 UI 文案调整，但仍建议至少做一次真实设备或模拟器验证。
- 最短手工验收路径至少包括：
  1. 启动应用
  2. 查看应用内用户可见名称位置
  3. 检查是否仍出现旧名
  4. 回归确认应用能正常启动
- 若当前环境不便完成设备验证，至少需要提供改动位置与启动验证结果。

## 对 Agent 的执行提示

- 先搜索 `creator-cfo-monorepo` 在 UI 层的出现位置，再判断哪些是真正展示给用户的文案。
- 不要一上来全局替换；只修改 UI 可见层。
- 发现属于技术配置、工程名、包名、slug 的字段时，直接跳过并在说明中标注“非本次范围”。
- 修改后至少做一次基础验证，避免只是替换了文案但没有确认页面闭环。
