# feat: 解析上传新增相机拍照入口

**公用约定**：以 `.cursor/prd/agent-dev-guide-summary.md` 为准。仓库级 Agent 基线（Cursor 始终应用）：`.cursor/rules/creator-cfo-always.mdc`。

## 平台描述

- 当前项目为 **Expo + React Native + Expo Router** 的移动端本地优先产品。
- 当前解析主链路已支持用户从既有上传入口选择文件并进入 parse / mapping / review 流程。
- 本需求的目标是：在**不引入后端、不重构既有解析主流程**的前提下，为解析页面新增**相机拍照入口**。

## 背景与目标

- 真实使用场景中，用户经常是拿到纸质票据后立即拍照上传，而不是先保存到相册或文件系统后再导入。
- 如果需求只写成“新增相机入口”，实现时很容易遗漏：
  - 相机权限申请与拒绝后的提示
  - 拍照结果是否真正进入现有解析流程
  - 拍照入口是否只做了 UI，没有打通主链路
  - 自动化测试与真实设备点击验收是否同步补齐
- 本 PRD 的目标是让 agent 明确理解：这不是“加一个按钮”，而是“新增拍照采集入口，并把拍到的照片稳定送入现有解析主链路”。

## 存储与契约影响

- **结构化 DB / 文件仓**：默认**不新增 SQLite 表**、**不新增文件仓集合**。
- **设备态 / 本地状态**：默认**不新增设置项**、**不新增持久化字段**。
- **须同步评估的工件**：
  - [ ] `packages/storage/src/contracts.ts`
  - [ ] `docs/contracts/`
  - [ ] `packages/schemas/src/index.ts`
  - [x] 自动化测试
  - [x] `tests/smoke/README.md`
- **说明**：
  - 如果本次只是新增拍照入口，并将拍照结果转成既有上传候选项，通常不需要改 contract/schema。
  - 如果拍照入口引入新的文件元数据语义、上传来源枚举、`sourceLabel` 或其他持久化字段语义变化，则必须同步更新 contract/schema/文档。

## 一句话需求

- 在解析页面新增相机拍照入口，用户拍照后，照片能够直接进入既有解析流程，后续 mapping / planner / review 逻辑保持不变。

## 需求描述

### 1. 解析页面新增相机入口

- 解析页面需要新增一个明确的相机拍照入口。
- 入口要求：
  - 用户易发现，不应隐藏在不明显位置
  - 命名和图标语义清晰，例如“拍照上传”“相机”
  - 与既有上传入口并存，不替代已有文件/相册入口
- 当前需求的重点是**新增入口**，不是重做整个上传页面。

### 2. 拍照结果进入既有解析链路

- 用户完成拍照后，照片必须进入现有解析链路。
- 对业务流程的要求：
  - 拍照结果应被视为一种上传输入来源
  - 后续 parse / mapping / planner / review / approval / persist 保持既有行为
  - 不允许因为拍照入口而新建一套平行的“相机专用解析逻辑”

### 3. 权限与异常处理

- 首次使用相机时，需要正确处理权限申请。
- 至少需要覆盖以下场景：
  - 用户授权相机权限后，可正常拍照
  - 用户拒绝权限后，得到清晰提示
  - 用户进入相机后取消拍照，不应导致页面异常或脏状态
  - 拍照结果为空或读取失败时，应有明确错误提示
- 不允许出现：
  - 点击相机按钮无反应
  - 权限被拒绝后无说明
  - 拍照取消后页面卡死或残留错误状态

### 4. 与现有上传能力的关系

- 本需求是对现有上传能力的补充，不是替换。
- 现有文件选择、相册选择、文档上传能力不应因本次改动被破坏。
- 如果拍照结果底层仍依赖图片上传逻辑，应优先复用已有实现，避免重复造轮子。

## 范围边界

**包含**：

- 解析页面新增相机拍照入口
- 相机权限申请与拒绝处理
- 拍照结果进入既有解析流程
- 自动化测试补齐相机场景或入口行为
- smoke 路径补齐真实拍照/模拟拍照验收步骤

**不包含**：

- 引入后端图片处理服务
- 新增独立相册管理、图片编辑、裁剪工作流
- 重构 mapping / planner / review 主业务逻辑
- 重构本地数据库、文件仓或非上传相关 UI
- 借此需求顺手改动其他上传入口的信息架构

## 实现约束

- 必须遵守 **mobile-first / local-first**，不能因为新增相机入口而引入后端依赖。
- 必须优先复用已有图片上传/解析逻辑，不要复制一套“相机专用” parse 流程。
- 若 Native 和 Web 平台能力不一致，必须明确：
  - 哪些平台支持真实相机
  - 哪些平台只能降级或不支持
  - 测试与 smoke 如何分别记录
- 优先做最小闭环：
  - 相机入口
  - 权限处理
  - 拍照结果接入既有链路
  - 验证证据

## 验收标准

- [ ] 解析页面存在清晰可点击的相机拍照入口。
- [ ] 用户授权相机权限后，可以完成拍照并进入既有解析流程。
- [ ] 用户拒绝相机权限时，会收到清晰可读的提示。
- [ ] 用户取消拍照时，页面不会报错、卡死或进入错误状态。
- [ ] 拍照上传后的后续 mapping / planner / review 主流程保持可用。
- [ ] 既有文件/相册上传能力没有因本次改动回归。
- [ ] 至少新增或更新 1 条自动化测试，覆盖相机入口或拍照接入的关键行为。
- [ ] `tests/smoke/README.md` 已补充“通过相机拍照进入解析”的手工验收路径。
- [ ] 如果没有 contract/schema 变更，PR 中需显式说明“本次仅新增相机入口，无 contract 变化”。

## 建议测试点

- 点击相机入口后，是否正确触发相机能力。
- 权限首次申请、拒绝、再次进入时的行为是否符合预期。
- 拍照完成后，上传候选项的文件名、mimeType、来源类型是否合理。
- 拍照结果是否能正常进入 parse，并继续到 mapping / review。
- 拍照取消后，页面状态是否恢复正常。
- 本次改动后，既有图片/文档上传是否没有回归。

## Testor 验收补充

- 本需求属于**移动端交互 + 设备能力调用 + 解析链路**需求，必须有真实设备或模拟器点击验收，不能只看自动化测试。
- 最短手工验收路径至少包括：
  1. 进入解析页面
  2. 点击相机入口
  3. 首次授权或验证权限提示
  4. 拍摄一张照片并确认使用
  5. 确认照片进入既有解析流程
  6. 再执行一次取消拍照，确认页面状态正常
  7. 回归一次原有文件/相册上传，确认未被改坏
- 若模拟器无法稳定提供拍照材料，可说明使用真实设备或使用模拟器图片注入方式，但需要在报告中注明环境。

## 对 Agent 的执行提示

- 先确认当前解析页面的上传入口结构，再决定把相机入口放在哪一层。
- 先查已有图片上传逻辑能否直接复用，避免在相机入口里重新拼装第二套解析链路。
- 先明确 `ImagePicker` / 相机权限 的现有使用方式，再做最小修改。
- 优先补“拍照成功进入解析”或“权限拒绝提示”的自动化测试，再补 smoke 路径，避免只有 UI 改动没有验证证据。
