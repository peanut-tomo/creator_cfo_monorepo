# Context Snapshot

## Version

- version: 0.5.0
- updated_at: 2026-04-07
- scope: refactor upload parsing into a planner-mediated workflow with PDF support, approval-gated counterparty/record writes, and workflow persistence

## Active Decisions

- Monorepo remains the baseline.
- Frontend stays on Expo + React Native + Expo Router.
- There is still no backend in the current phase.
- Upload remains local-first with SQLite plus the on-device file vault.
- Owner entity stays fixed to `entity-main` for this slice.
- `source` and `target` resolve only through local `counterparties`.
- OpenAI parser output is never treated as final bookkeeping truth without local validation and approval.
- `originData` remains the persisted field name for compatibility, but now stores a validated parser DTO snapshot rather than arbitrary JSON.
- `scheme` is deprecated compatibility output and no longer participates in the active parser/planner workflow.

## Implemented Structure

- `packages/storage/src/contracts.ts`: contract v5 with upload/planner workflow tables, planner payload persistence, and indexes.
- `packages/schemas/src/index.ts`: strict `ReceiptParsePayload`, strict `ReceiptPlannerPayload`, workflow enums, and local projection helpers.
- `apps/mobile/src/features/ledger/*`: workflow-aware native/web runtimes, strict parser/planner validation, planner review UI, and approval helpers.
- `docs/upload-planner-workflow.md`: workflow spec for parser -> planner -> local validation -> approval -> final persistence.
- `.cursor/skills/project/receipt-parse` and `.cursor/skills/project/receipt-db-update-planner`: project-local prompt skills mirrored in runtime prompts.
