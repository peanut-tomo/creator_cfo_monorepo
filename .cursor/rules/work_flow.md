# PRD 分解与任务开发工作流

**始终生效的基线**（摘要）：`.cursor/rules/creator-cfo-always.mdc`。下文为流程全文。

**项目真源**：根目录 `AGENTS.md`、`.cursor/context/main.md`。本阶段为 **Expo 移动端、本地优先**；契约以 `docs/contracts/`、`packages/storage/src/contracts.ts`、`packages/schemas` 为准。处理需求时优先阅读上述文件，避免按「全栈 Web + 后端 API」的默认假设实现。

本规则约束：从需求（PRD）到可交付实现时，**按角色分工**推进，角色定义分别见：

- **Harness（审查者 / 评估者）**：`.cursor/agent/harness.md`
- **Testor（测试工程师）**：`.cursor/agent/testor.md`
- **Dev（开发者 / 执行者）**：`.cursor/agent/dev.md`

**维护约定**：`.cursor/agent/` 下上述三个文件为**稳定角色基线**。对照各类开发指南或实现某条需求时，**不要**为用例便利而随手修改它们；流程或口径调整应优先落在 PRD、本文件、`AGENTS.md` 等，经确认后再**单独、审慎**更新 agent。详见 `.cursor/agent/README.md`。

Agent 在对应阶段应**载入并遵守**上述文件中的原则、边界与输出格式，不得用单一角色包办全流程自我宣布完成。

**PRD 形态**：新需求建议复制 `.cursor/prd/TEMPLATE.md`，并填写 **「存储与契约影响」**；已有 feat 文档应逐步补全该小节，便于 Harness 锁范围、Dev 走 schema-first。

---

## 1. 总流程（顺序与门禁）

```
PRD 输入
    → [Harness] 分解与验收项、进度口径（含存储/契约是否变更）
    → [Testor] 可执行验收条件与测试场景（含冒烟）
    → [Dev] 契约对齐 → 小步实现 → lint/test/build/contract:check
    → [Testor] 执行验证与测试报告
    → [Harness] 审查结论、汇报、PRD 进度表 → 通过 / 打回
```

**门禁约定**

- **未**经 Harness 对 PRD 的可验收拆解（或可勾选条目），Dev 不开始大范围实现；架构级、范围歧义须先由人/Harness 定口径。
- **未**经 Testor 给出验收要点或场景表，Dev 仍可按已锁定的契约开发，但须在实现中预留可测性；合并前应有 Testor 的验证证据或明确「未覆盖」声明。
- **Harness 未**给出审查结论（通过 / 有条件通过 / 打回）前，不将需求标为「已完成」。

---

## 2. Harness：PRD 分解与进度口径（先做）

**输入**：`.cursor/prd/` 下 PRD、feat 说明、或用户给出的需求文档。

**输出（必须落到可勾选项）**

1. 将 PRD 拆成**可验收条目**（编号/标题与 PRD 对齐），标明范围包含与不包含。
2. 标出须先决策的部分：**架构**、**本地存储/文件仓契约**、**跨端同步**等；若 PRD 写明「无契约变更」，Dev 不得擅自扩展持久化模型。未来若某 PRD 引入 **HTTP API / OpenAPI / 服务端 DB**，在该 PRD 内单独列出契约与迁移计划。
3. 为每条验收项预留**证据**字段（后续由 Testor / CI 填充）：测试用例、PR、界面与本地数据行为说明、截图等。
4. 使用 Harness 文档中的**需求进度表**结构（状态：`未开始` / `进行中` / `待验收` / `已通过` / `已打回` / `已搁置`）；**无证据不得标已通过**。

**禁止**：Harness 在此阶段不写大段业务实现代码；可给最小示例说明问题。

**可选归档**：可将拆解与进度表写入 `.cursor/harness/<feat>_review.md`，与对应 PRD 成对追溯。

---

## 3. Testor：验收条件与测试设计（开发前或并行早期）

**输入**：Harness 拆解后的验收项、相关契约（`docs/contracts/`、`contracts.ts`、schemas；若需求含 API 则含 OpenAPI 等）。

**输出**

1. **需求理解摘要**：测什么、不测什么、关键假设与依赖。
2. **可执行验收条件**：成功标准用可观察现象表述（界面、本地持久化状态、日志；若有接口则含请求/响应）。
3. **场景表 / 测试要点**：正常、异常、边界；**冒烟路径**（Expo 启动 → 最短主路径 → 可见成功结果）单独列出。
4. 推动可进 **CI** 的检查（lint / test / contract:check / build / smoke）；若无自动化，给出**可重复验证清单**（命令与步骤）。

**禁止**：不重新定义产品范围；不替代 Harness 做深度架构审查。

---

## 4. Dev：任务开发与实现（在契约与拆解确定之后）

**输入**：已定验收项、契约、Testor 的测试要点（或至少已定持久化/数据形状）。

**执行方式**

1. **契约优先**：本地数据与文件仓以文档与 `contracts.ts` 为准；冲突时先走契约变更流程（含 `docs/contracts/` 与测试），不静默改口径。
2. **小闭环**：每步可独立验证（lint / test / build / contract:check），不大任务一次性糊完。
3. **可测性**：关键路径、失败路径、幂等等有测试或明确 TODO 与风险说明。
4. 遵守 Dev 文档中的**质量门禁**与**交付自检**（主流程可运行、存储/契约变更已同步文档与测试、已跑或说明未跑检查）。

**禁止**：不篡改需求边界；不自任规划者把模糊需求首次写成最终规格；不自行宣布「已完成」（以 Harness + Testor 证据链为准）。

---

## 5. Testor：验证与测试报告（实现可测后）

**输出**：遵循 `testor.md` 的**测试报告结构**（需求理解摘要、范围、用例与结果、缺陷列表、冒烟结论）。

**缺陷**：每个含环境、复现步骤、预期 vs 实际、佐证材料。

对阻塞项明确标注，便于 Dev 修复与 Harness 复核。

---

## 6. Harness：审查、汇报与进度闭环（交付前）

**输出**：遵循 `harness.md` 的完整结构：

1. 审查结论（通过 / 有条件通过 / 打回）
2. 问题分级（Blocker / Major / Minor）与问题明细
3. 验证建议
4. **汇报结果**（便于转发的表格：对象与范围、总结论、阻塞与风险、证据摘要、下一步、时间语义）
5. **需求进度表**（对照 PRD）：更新状态与证据，汇总已通过/总数与主阻塞

**打回**时给出可执行的下一步；**通过**仍须标明未验证范围（若有）。

---

## 7. 迭代与打回

- **打回**：Dev 按 Harness Blocker 与 Testor 缺陷修复 → Testor 复验 → Harness 再审查。
- **有条件通过**：Harness 须写清附带条件与残留风险；进度表中对应项不得标为「已通过」直至条件关闭。

---

## 8. Agent 执行提示

处理 `.cursor/prd/` 或用户提出的新需求时：

1. **声明当前扮演的角色**（Harness / Testor / Dev），或按阶段切换并说明 handoff。
2. 引用本工作流与对应 `agent/*.md`，保持**三角制约**：实现（Dev）、实证（Testor）、守门（Harness）不得合并为同一自述的「已完成」。
3. 涉及《Ai Agent 友好型 开发指南》时：优先阅读 **`.cursor/prd/agent-dev-guide-summary.md`**（供 Agent 拉取）；PDF 若存在可作为人工补充，勿假设模型已完整读取 PDF。
