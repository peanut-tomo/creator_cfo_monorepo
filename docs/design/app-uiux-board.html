Repository Analysis + App UI/UX

CreatorCFO 移动端产品骨架与 UI/UX 设计稿

这个仓库当前已经明确了产品的几个关键前提：`移动端优先`、`本地优先`、`四个底部 Tab`、`Expo + SQLite + File Vault + AsyncStorage`。 但业务页面仍处于壳层阶段，所以这版设计图的目标不是复刻现状，而是**在不违背当前架构的前提下，把它推进成真实可用的 Creator 财务 App**。

当前形态 Expo Router + React Native 本地优先 App
现有 IA Login / Home / Ledger / Discover / Profile
设计目标 让壳层变成真正的创作者财务工作流

分析结论

仓库当前做到了什么

已经搭好了登录、四个 Tab、本地会话、主题、多语言和本地存储契约。也就是说，技术底座已经具备继续往真实产品演进的条件。

主要缺口

还没进入业务深水区

Home 更像项目介绍页，Ledger / Discover / Profile 仍偏占位。缺少真正的收入、对账、税务和助手场景界面。

设计策略

保留现有 IA，重做内容层

这版设计不推翻四 Tab 结构，而是在 Home、Ledger、Discover 中放入真正的 CreatorCFO 工作流，并把 Tax Assistant 作为 Discover 的重点入口。

Design Reasoning

为什么这样设计

不推翻当前四 Tab 结构

仓库已经把导航、状态、主题和会话模型定下来了，UI 最好沿着这个外壳继续深化。

首页承担决策，不承担介绍

Home 应该回答“我今天财务上最该看什么”，而不是展示系统模块说明。

Tax Assistant 最适合放在 Discover

从当前 IA 看，Discover 是承接 AI、模板、指南和扩展能力的最佳位置。

Product Translation

从仓库到产品的映射

  • HomeScreen 现状是项目概览，建议升级成 cash + reserve + tasks dashboard。
  • LedgerScreen 现状是占位卡片，建议升级成收入 / 费用 / 发票混合工作台。
  • DiscoverScreen 现状最轻，最适合承接 Tax Assistant 和模板资料库。
  • ProfileScreen 当前结构合理，保持主题 / 语言 / 会话管理即可。

Next Screens

下一步最值得细化的页面

  • Tax Assistant 详细流程页
  • Receipt capture / file vault 页面
  • Invoice detail / AR tracking 页面
  • Month-end review 页面
  • Settings 中的 local vault 管理页
这版 UI/UX 设计图是基于当前仓库的真实结构做的“产品化推进稿”，不是脱离代码的概念稿。核心判断是：这个仓库已经具备移动端本地优先的产品底座，接下来最该补的是业务工作流，而不是继续扩展说明型页面。
