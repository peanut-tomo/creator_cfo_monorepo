# CreatorCFO 一期记账版设计图构思 PRD

## 1. 文档目的

本文档用于统一 CreatorCFO 一期移动端设计图的构思方向，帮助产品、设计、研发在同一个边界下推进页面设计与原型输出。

这份文档解决的问题不是“功能是否要做”，而是：

1. 当前版本设计图应该围绕什么核心目标展开
2. 哪些页面需要重点设计
3. 各页面应该承接什么任务
4. 视觉和交互上应该强调什么
5. 哪些内容本期明确不做，避免设计发散

---

## 2. 项目背景

结合当前仓库形态，CreatorCFO 目前是一个：

- 移动端优先
- 本地优先
- 基于 Expo + React Native 的 Creator Finance App

当前一期不做完整财税产品，只聚焦最基础、最刚需的轻量工作流：

- 记账
- 上传
- 解析

也就是说，本期不是做完整的税务产品，也不是做自动同步平台的复杂 SaaS，而是先把创作者在手机端整理经营记录的基础流程做通。

---

## 3. 一期产品范围

### 3.1 本期要做

一期只做以下三类能力：

1. 记账
- 记录收入
- 记录费用
- 记录发票

2. 上传
- 上传 receipt
- 上传 statement
- 上传 invoice file
- 上传 support doc

3. 解析
- 对上传后的文件做基础字段识别
- 让用户确认解析结果
- 把解析结果关联到具体记录

### 3.2 本期不做

以下内容本期不进入设计主线：

1. 不做税务计算
2. 不做 Tax Assistant 主流程
3. 不做银行自动同步
4. 不做平台自动接入
5. 不做正式报税
6. 不做新的底部菜单
7. 不做复杂会计、对账或多端协同

---

## 4. 设计前提与约束

### 4.1 底部菜单不改动

这是本期最重要的设计约束。

当前底部菜单保持不变：

1. `Home`
2. `Ledger`
3. `Discover`
4. `Profile`

本期所有新能力都必须放入这四个既有菜单及其二级页面中承接，不能新增新的底部 Tab。

### 4.2 新能力只做页面内增强

设计时要坚持：

- 在 `Home` 增加财务总览和待处理提醒
- 在 `Ledger` 增加记账和上传入口
- 在 `Discover` 增加工具和资料入口
- 在 `Profile` 保持轻量，不扩展业务主流程

### 4.3 符合当前仓库阶段

设计图必须符合当前项目现实，而不是设计成一个已经有完整后端和自动同步能力的大型财务 SaaS。

因此设计上要体现：

- 本地优先
- 移动端优先
- 工作流清晰
- 页面可落地

---

## 5. 产品目标

### 5.1 本期产品目标

让用户在不离开手机的情况下完成以下最小闭环：

1. 新增一笔收入、费用或发票记录
2. 上传相关文件
3. 查看并确认解析结果
4. 将文件和记录关联起来
5. 在首页和账本页看到自己的最新财务状态

### 5.2 本期设计目标

设计图要帮助用户清楚理解：

1. 这个产品当前是“创作者记账工作台”
2. 上传和解析是记账的辅助流程，不是独立系统
3. 现有菜单已经足够承接一期功能
4. 首页、账本页、发现页各自职责清晰

---

## 6. 目标用户

本期设计服务的用户是：

- 创作者
- 自雇人士
- 自由职业者
- 轻量经营者

这类用户通常有以下特点：

1. 财务记录零散
2. 收入、费用、发票资料分散在手机和文件里
3. 不喜欢复杂会计软件
4. 需要一个更轻量、更直观的整理工具

因此设计上要避免：

- 过度专业化的会计界面
- 复杂表格型页面
- 企业 ERP 风格

---

## 7. 产品定位

一期的产品定位应该明确为：

**一个移动端、本地优先的创作者记账与资料整理工具。**

不是：

- 银行 App
- 预算管理 App
- 通用会计软件
- 报税工具

设计上应让用户感觉：

“这是一个帮助我整理经营记录和资料的工作台。”

---

## 8. 设计原则

### 8.1 不改结构，先补能力

优先在现有结构里增加能力，不通过新增菜单来解决问题。

### 8.2 先让流程通，再追求复杂度

本期更重要的是让“记账 -> 上传 -> 解析 -> 关联”这条链路成立，而不是页面做得特别复杂。

### 8.3 上传和解析必须服务记账

上传和解析不应变成独立入口产品，而应明确从属于记账主流程。

### 8.4 移动端要可快速完成动作

每个页面都应有明确主动作，不做大而散的信息堆叠。

### 8.5 界面要像真实产品，而不是概念海报

设计图需要具备：

- 清晰层级
- 可操作按钮
- 真实底部导航
- 合理卡片结构
- 真实移动端布局

---

## 9. 信息架构

## 9.1 Home

### 页面角色
财务首页 / 今日总览

### 页面职责

1. 展示当前财务状态
2. 展示待处理事项
3. 作为进入 Ledger 和 Upload 的起点

### 本期应新增的能力

1. 财务摘要卡
2. 今日待处理卡
3. 最近上传状态
4. 待解析提醒

### 不应承载的内容

1. 大量静态产品说明
2. 复杂设置项
3. 税务主流程

---

## 9.2 Ledger

### 页面角色
主记账工作区

### 页面职责

1. 展示收入、费用、发票记录
2. 提供新增记录入口
3. 提供上传入口
4. 承接详情查看和编辑

### 本期应新增的能力

1. `Income / Expenses / Invoices` 分段切换
2. `New Record` 入口
3. `Upload` 入口
4. 最近记录列表
5. 记录状态
6. 详情页跳转

### 页面定位

Ledger 是本期最核心的功能页面。

---

## 9.3 Discover

### 页面角色
工具与资料入口页

### 页面职责

1. 展示上传历史
2. 展示解析工具入口
3. 展示模板或资料入口
4. 承接“下一步要做什么”的工具性引导

### 本期应新增的能力

1. Upload history
2. Parse Review 入口
3. Template vault
4. 文件相关帮助说明

### 不应承载的内容

1. 资讯流
2. 社区内容
3. 完整税务系统

---

## 9.4 Profile

### 页面角色
偏好和会话页

### 页面职责

1. 保持现有主题 / 语言 / 会话管理能力
2. 不承担本期主业务流程

### 设计要求

Profile 可以保持轻量，不作为本期重点设计页。

---

## 10. 关键二级页面

在不新增底部菜单的前提下，本期设计图应补充以下二级页面。

### 10.1 Upload Center

#### 作用
统一承接文件上传入口

#### 支持类型

1. Receipt
2. Statement
3. Invoice file
4. Support doc

#### 页面目标

1. 让用户快速选择文件类型
2. 完成上传
3. 明确知道上传后会进入解析或关联流程

---

### 10.2 Parse Review

#### 作用
承接上传后的解析确认流程

#### 页面目标

1. 展示系统识别出的字段
2. 让用户确认或修改
3. 决定是新建记录还是关联已有记录

#### 核心认知

Parse Review 不是独立菜单，而是上传后的工作流页面。

---

### 10.3 Record Detail

#### 作用
查看和编辑单条记录

#### 页面目标

1. 查看核心字段
2. 查看当前状态
3. 查看关联文件
4. 追加附件
5. 更新记录状态

---

## 11. 核心用户流程

### 流程 1：新增收入记录

1. 用户进入 `Ledger`
2. 点击 `New Record`
3. 选择 `New Income`
4. 填写字段
5. 可选上传 statement 或 proof
6. 保存记录

### 流程 2：新增费用记录并上传票据

1. 用户进入 `Ledger`
2. 点击 `Upload`
3. 选择上传 `Receipt`
4. 进入 `Parse Review`
5. 确认金额、分类和日期
6. 生成新的 expense record

### 流程 3：查看待处理文件

1. 用户进入 `Home`
2. 看到待解析提醒
3. 点击进入 `Parse Review`
4. 确认字段
5. 关联到具体记录

### 流程 4：查看记录与附件

1. 用户进入 `Ledger`
2. 打开某条记录
3. 进入 `Record Detail`
4. 查看已关联文件
5. 追加新文件或更新状态

---

## 12. 页面设计重点

## 12.1 Home 设计重点

设计图上应重点体现：

1. 财务概览
2. 待处理事项
3. 最近上传 / 待解析状态
4. 清晰进入 Ledger 的路径

Home 不应像说明页，而应像工作台首页。

---

## 12.2 Ledger 设计重点

Ledger 是本期核心页面，设计图上应重点体现：

1. 分段切换
2. 记录列表
3. 新增入口
4. 上传入口
5. 记录状态

这页应该最有“业务工作区”的感觉。

---

## 12.3 Upload Center 设计重点

应重点体现：

1. 文件类型清晰
2. 操作简单
3. 上传之后会进入下一步
4. 不像系统文件管理器

---

## 12.4 Parse Review 设计重点

应重点体现：

1. 解析字段结构清晰
2. 用户可以改、确认、跳过
3. 能明确关联到记录
4. 页面像“确认工作流”，不是聊天页

---

## 12.5 Record Detail 设计重点

应重点体现：

1. 详情清晰
2. 状态清晰
3. 附件可见
4. 可继续操作

---

## 13. 视觉方向建议

设计图应采用以下视觉方向：

### 13.1 整体气质

- 专业
- 清晰
- 冷静
- 可信
- 移动端原生感强

### 13.2 视觉风格

- 浅暖色背景
- 深蓝色正文
- 低饱和绿色强调色
- 大圆角卡片
- 清晰的卡片分层

### 13.3 避免的风格

1. 消费记账 App 风格
2. 银行 App 风格
3. 加密交易所风格
4. 企业 ERP 风格
5. 过度未来感的概念设计

---

1. `Home`
2. `Ledger`
3. `Upload Center`
4. `Parse Review`
5. `Discover`
6. `Record Detail`

建议分两层输出：

### 第一层：主流程页

1. Home
2. Ledger
3. Upload Center
4. Parse Review

### 第二层：辅助页

1. Discover
2. Record Detail
3. New Income
4. New Expense
5. New Invoice

---

## 15. 验收标准

如果一版设计图满足以下条件，则可认为构思方向正确：

1. 底部菜单没有变化
2. 用户可以清楚看出本期只做记账、上传、解析
3. Ledger 被明确识别为主工作区
4. 上传和解析是明确的页面流，不是抽象概念
5. Home 不再只是说明页
6. Discover 没有被设计成无关内容页
7. 页面间关系清晰，逻辑自然
8. 界面像真实产品而不是海报

---

## 16. 后续设计延展建议

在这版构思确认后，下一步建议继续细化：

1. `New Income` 高保真表单页
2. `New Expense` 高保真表单页
3. `New Invoice` 高保真表单页
4. 上传成功态 / 失败态
5. 解析失败态 / 低置信度态
6. 空状态页面
7. Record Detail 的编辑态

---

## 17. 结论

CreatorCFO 一期设计图的核心不是扩展功能，而是：

**在不改原有菜单的前提下，把记账、上传、解析这三件事自然放进现有移动端结构中。**

所以本期设计图最重要的任务是：

1. 让 `Home` 更像工作台
2. 让 `Ledger` 成为主记账中心
3. 让 `Upload -> Parse -> Link Record` 成为清晰流程
4. 让 `Discover` 成为轻工具入口，而不是新业务线

这套构思适合继续往：

- 高保真 UI
- React Native 页面实现
- 页面级交互说明

三个方向继续推进。
